# Документация по удалённому подключению к ComfyUI и запуску воркфлоу

## 1. Предварительные требования
- **Сервер с GPU** (Linux / Windows) где будет установлен **ComfyUI**.
- **Python 3.10+** и возможность запускать `python main.py`.
- Открытый порт **8188** (по умолчанию) или любой другой, который будет использоваться для API.
- Доступ к серверу по HTTP/HTTPS из вашего основного окружения (OpenClaw).
- Установлен **ComfyUI‑Remote‑Executor** (расширение, позволяющее отправлять воркфлоу на удалённый сервер и получать результаты через веб‑хук).

## 2. Установка ComfyUI и Remote‑Executor
```bash
# На удалённом сервере
cd ~
git clone https://github.com/comfyanonymous/ComfyUI.git
cd ComfyUI
# установить зависимости
python -m pip install -r requirements.txt

# установить расширение Remote‑Executor
cd custom_nodes
git clone https://github.com/cksdnfas/ComfyUI-Remote-Executor.git
```
После установки перезапустите ComfyUI.

## 3. Запуск ComfyUI с поддержкой CORS
Для того, чтобы запросы из вашего основного сервера проходили без блокировок, запустите ComfyUI с параметрами:
```bash
# Linux
python main.py --listen 0.0.0.0 --enable-cors-header

# Windows (standalone build)
./python_embeded/python.exe -s ComfyUI/main.py --windows-standalone-build --listen 0.0.0.0 --enable-cors-header
```
- `--listen 0.0.0.0` — делает API доступным извне.
- `--enable-cors-header` — разрешает кросс‑доменные запросы, нужные для удалённого вызова.

> **Важно:** проверьте, что firewall пропускает входящие соединения на указанный порт.

## 4. Настройка Remote‑Executor в UI
1. Откройте ComfyUI в браузере (`http://<SERVER_IP>:8188`).
2. Нажмите на шестерёнку ⚙️ → **Remote Servers**.
3. Добавьте новый сервер:
   - **Name:** произвольное название, например `GPU‑Workstation`.
   - **URL:** `http://<SERVER_IP>:8188`
   - **Authorization (optional):** Bearer‑токен или API‑ключ, если вы захотите ограничить доступ.
4. Сохраните.
5. (Опционально) включите **Auto Upload** и укажите URL вашего веб‑хука, куда будут отправляться готовые изображения.

## 5. API‑вызов для выполнения воркфлоу
ComfyUI имеет два основных эндпоинта:
- `POST /prompt` — принимает JSON с описанием графа (workflow) и возвращает `prompt_id`.
- `GET /history?prompt_id=<id>` — возвращает статус выполнения и ссылки на сгенерированные файлы.

### Пример тела запроса `/prompt`
```json
{
  "prompt": {
    "3": {"class_type": "LoadImage", "inputs": {"image": "path/to/input.jpg"}},
    "4": {"class_type": "KSampler", "inputs": {"seed": 12345, "steps": 20, "cfg": 7.0, "sampler_name": "euler", "model": "sd15"}},
    "5": {"class_type": "SaveImage", "inputs": {"images": ["4"]}}
  }
}
```
Ответ:
```json
{"prompt_id": "a1b2c3d4"}
```
### Получение результата
```bash
curl "http://<SERVER_IP>:8188/history?prompt_id=a1b2c3d4"
```
В ответе будет массив `output` с путями к файлам (обычно в `output/`).

## 6. Автоматическая отправка изображения в Telegram
1. **Веб‑хук**: создайте простой HTTP‑endpoint (например, Flask) который принимает multipart‑form‑data и сохраняет файл или сразу отправляет его в Telegram.
2. **Telegram‑клиент**: для публикации от имени аккаунта (не бота) нужен *user‑client* (TDLib / Telethon). Бот‑API не позволяет публиковать от имени обычного пользователя.
   - Сгенерируйте `api_id` и `api_hash` в [my.telegram.org](https://my.telegram.org).
   - Авторизуйтесь один раз, получив `session_string` (можно сохранить в файле).
   - Используйте `client.send_file` с параметром `caption` – сюда вставляем описание рекламного креатива.
3. Веб‑хук после получения изображения сразу вызываете функцию отправки в Telegram.

## 7. Примерный рабочий поток
1. **Cron‑задача** (OpenClaw) формирует текстовый промпт (рекламный лозунг, детали продукта).
2. Cron вызывает *systemEvent* → ваш агент собирает JSON‑описание воркфлоу (с нужными узлами, в том числе `LoadPrompt`/`KSampler`/`SaveImage`).
3. Агент отправляет запрос `POST /prompt` к удалённому ComfyUI.
4. Получив `prompt_id`, агент периодически (или через webhook Remote‑Executor) проверяет статус.
5. Когда изображение готово, оно автоматически POST‑ится на ваш веб‑хук.
6. Веб‑хук сохраняет файл и через Telegram‑user‑client публикует его в выбранный канал.
7. При необходимости сохраняем метаданные (prompt, ссылка, дата) в базе/файле для отчётов.

## 8. Возможные «дыры» и риски
- **Доступ к Telegram от имени пользователя** – требуется хранить `session_string` и соблюдать правила Telegram (может быть блокировка за спам).
- **Безопасность удалённого сервера** – открытый порт 8188 должен быть защищён (firewall, VPN, авторизация в Remote‑Executor).
- **Скорость и очередь** – если генерировать много рекламных креативов, GPU может стать узким местом; стоит добавить несколько серверов и распределять нагрузку.
- **Контент‑модерация** – ИИ‑генерация может выдавать NSFW‑изображения; рекомендуется добавить проверку (например, с помощью CLIP‑модели) перед публикацией.
- **Стабильность webhook** – если endpoint недоступен, изображения потеряются; реализуйте очередь / повторные попытки.
- **Лимиты Telegram** – ограничение на количество сообщений в канал в сутки (~20‑30 мб/сообщение), а также правила по рекламному контенту.

## 9. Оценка сложности
| Шаг | Трудоёмкость | Примечание |
|-----|---------------|------------|
| Подготовка удалённого сервера (GPU, ComfyUI, CORS) | Средняя (2‑3 ч) | Нужно установить драйверы, Docker/venv, протестировать доступность. |
| Установка Remote‑Executor и настройка webhook | Низкая‑средняя (1‑2 ч) | Веб‑хук простая Flask‑приложение. |
| Разработка cron‑задач в OpenClaw | Низкая (30 мин) | Описание JSON‑prompt, парсинг ответа. |
| Telegram user‑client интеграция | Средняя‑высокая (3‑5 ч) | Авторизация, хранение сессии, обработка ошибок. |
| Модерация и логирование | Средняя (2 ч) | Добавить проверку контента и запись в `memory/`.
| Тестирование и отладка | Средняя‑высокая (4‑6 ч) | Сценарии падения, повторные попытки.
**Итого:** примерно **12‑20 часов** разработки при наличии готового GPU‑сервера.

## 10. Выводы
- План реализуем и технически выполним, главное правильно обеспечить безопасный удалённый доступ к ComfyUI и надёжную публикацию в Telegram от пользовательского аккаунта.
- Ключевые задачи – настройка CORS, веб‑хук и Telegram‑client, а также автоматическая проверка качества контента.
- После первого прототипа дальнейшее масштабирование (добавление серверов, очередь задач) будет простым, так как все части уже разбиты на независимые сервисы.
